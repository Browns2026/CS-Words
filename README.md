#### 1. HashCode
- 21 April, 2026
    - HashCode란 객체를 빠르게 찾기 위한 정수값(Hash Value)이다.

    예를 들어, "Apple".GetHashCode()
    를 실행하면 문자열 "Apple"에 대한 정수값이 반환된다.
    
    
    컴퓨터에서 Apple 이라는 이름의 데이터를 하나씩 비교하며 검색하는 Linear Search(순차 탐색) 다소 속도가 느릴 수 있다는 단점이 있다. 따라서 HashCode 를 사용하면 특정 데이터를 정수값으로 변환해, 저장위치를 빠르게 찾을 수 있다.

    - 사용처

    Dictionary, HashSet, Cache(로그인 캐시), DB Indexing, Double Check(중복검사) 등에서 사용된다.

    - 중요한 특징
    "Apple" 이라는 두개의 값이 있을 때, 동일한 HashCode 를 반환한다. 하지만 서로 다른 값이 동일한 HashCode 를 가질 수도 있는데, 우리는 이를 Hash Collision (해시 충돌) 이라고 한다.

    - 결론

    HashCode 는 빠른 검색/분류를 위한 기술이지 절대적인 고유 ID 는 아니다.

        - 참고 : 자료구조, Dictionary 내부 동작, DB Indexing, Cache System 에서 더 자세히 다룬다.




#### 2.CPU와 GPU의 차이
- 22 April, 2026
    - CPU(Central Processing Unit)는 컴퓨터 전반적인 연산과 제어를 담당하는 장치다
    
    기본적으로 복잡한 계산이나 판단 작업에 강하며, 운영체게 실행, 프로그램 처리, 게임 로직, 조건문 처리 등을 담당한다.

    예를 들어, 게임의 AI, 충돌판정, 프로그램 실행, 인터넷 브라우저 동작 등은 대부분 CPU 가 처리한다.

    - GPU(Graphics Processing Unit)는 대량의 단순계산을 동시에 처리하는 장치이다.

    원래는 그래픽 처리를 위해 만들어졌지만, 현재는 영상처리나 AI 연산 등 다양한 분야에서도 사용되고 있다.
    
    예를들어, 게임 그래픽 출력, 그림자 처리, 조명효과, 렌더링, AI 딥러닝 연산 등은 GPU 에서 담당한다.


    - CPU와 GPU의 차이

    | CPU | GPU |
    | --- | --- |
    | 복잡한 계산 | 단순 반복 계산 |
    | 적은 수의 고성능 코어 사용 | 많은 수의 코어 사용 |
    | 판단 및 제어 중심 | 병렬 처리 중심 |
    | 운영체제와 프로그램 관리 | 그래픽 및 대량 연산 처리 |
    

    특히 게임 개발에서는 Unreal Engine 이나 Unity 와 같은 게임 엔진에서는 CPU 와 GPU 가 함께 동작한다.
    주로 Game Engine 에서 CPU 가 담당하는 작업으로는 게임로직, 플레이어 입력 처리, 몬스터 AI, 충돌 판정, 물리 계산 등이 있으며, 
    GPU가 담당하는 작업으로는 렌더링, 배경출력, 이펙트 처리, 조명 및 그림자, 애니메니션 화면 출력 등이 있다.

    - 결론

    CPU 는 복잡한 계산과 제어를 담당하는 장치이며, GPU 는 대량의, 그래픽 및 병렬 연산을 처리하는 장치이다.
    두 장치는 서로 역할이 다르며, 현대의 프로그램과 게임에서는 CPU와 GPU 가 함께 동작하여 성능을 향상시킨다.



#### 3. 소프트웨어의 정의
- 25 April, 2026
    - 소프트웨어(Software)란, 컴퓨터가 하드웨어를 동작시키기 위한 프러그렘과 데이터의 집합이다.

    즉, 사용자가 원하는 작업을 수행할 수 있도록 컴퓨터에게 명령을 전달하는 역할을 한다.
    따라서, 소프트웨어는 눈에 보이지 않는 논리적 구성 요소이며, 하드웨어 없이 단독으로 동작할 수 없다.

    - 하드웨어와 소프트웨어의 차이
    
    | Hardware | Software |
    | --- | --- |
    | 물리적으로 존재 | 비물리적 존재 |
    | 실제 만질 수 있음 | 코드와 데이터 형태로 존재 |
    | CPU, RAM, 키보드 등 | Windows, Unity, Game 등 |
    | 시간이 지나면 마모됨 | 수정과 업데이트 가능 |

    - 소프트웨어의 종류 2가지

    | 시스템 | 응용 |
    | --- | --- |
    | 컴퓨터 자체를 동작이시키 위한 소프트웨어 | 사용자가 특정 작업을 수행하기 위한 소프트웨어 |
    | Windows, MacOs, Linux | Unity, Chrome, Photoshop, Game Program |
    
    - 소프트웨어의 역할
    하드웨어 제어, 사용자 명령 처리, 데이터 저장/관리, 프로그램 실행, 사용자와 컴퓨터 간의 상호작용 제공 등이 있다.

    - 결론
    소프트웨어는 컴퓨터 하드웨어를 동작시키기 위한 프로그램과 데이터의 집합이며, 사용자가 원하는 기능을 수행할 수 있도록 컴퓨터를 제어하는 역할을 한다. 
    


#### 4. 소프트웨어의 특성
- 26 April, 2026
    - 소프트웨어는 하드웨어와 달리 물리적인 형태가 존재하지 않으며, 논리적인 구조로 이루어진 프로그램과 데이터의 집합니다.

    소프트웨어는 개발 이후에도 지속적으로 수정 및 유지보수가 이루어진다는 특징이 있다.

    - 주요 특성
    1. 복잡성 (Complexity)
    
    기능이 많아질수록 구조가 복잡해진다.
    - 코드의 양이 증가할 수록, 버그 발생 가능성의 증가, 유지보수 유지 어려움, 기능 간 의존성이 증가하는 등의 문제가 발생할 수 있다.
    
    2. 비가시성 (Invisibility)
    
    소프트웨어는 눈으로 직접 볼 수 없는 논리적 구조이다.
    - 하드웨어처럼 형태가 존재하지 않기에 내부 구조를 이해하기 어렵다.
    
    3. 변경 가능성 (Changeability)
    
    소프트웨어는 지속적인 수정과 업데이트가 가능하다.
    - 사용자의 오구사항 변화나 버그 수정에 따라 지속적으로 변경된다.

    4. 비마모성 (Non-Wear)

    하드웨어는 시간이 지나면 마모되지만, 소프트웨어는 물리적으로 닳지 않는다.
    - 하지만 잘못된 수정이나 유지보수로 인한 품질 저하 가 이러날 수 있다. 이는 1번 특성과 비슷하다.

    5. 복제 가능성 (Reproducibility)
    
    소프트웨어는 동일한 형태로 무한 복제가 가능하다.
    - 복사 과정에서도 품질 손실이 거의 발생하지 않는다.

    - 결론

    소프트웨어는 복잡성, 비가시성, 변경가능성, 복제가능성 등의 특징을 가지며, 이러한 특성으로 인한 지속적인 관리와 유지보수가 중요하다.

    


#### 5. 소프트웨어 위기
- 27 April, 2026
    - 소프트웨어 위기(Software Crisis)란, 컴퓨터 기술의 발전 속도에 비해 개발 기술과 관리능력이 따라가지 못하면서 발생한 문제를 의미한다.

    해당 개념은 1960년대 후반부터 대형 소프트웨어 프로젝트가 증가함에 따라 등장한 개념이다.

    - 원인

    컴퓨터 하드웨어 성능은 빠르게 발전했지만, 소프트웨어 개발 방식은 체계적이지 못했다.
    
    그로인한 결과, 프로젝트 규모, 코드의 복잡성, 개발 기간이 증가했고, 이에 따른 유지보수가 어려워졌다.

    - 주요 문제점
    1. 개발 일정 지연
    2. 비용 증가
    3. 낮은 품질
    4. 유지보수 어려움
    5. 생산성 저하
    
    이 있으며, 

    - 해결 방안
    소프트웨어공학(Software Engineering) 이 등장하게 되었다.
    
    이를 통해 체계적인 개발 방법, 설계 기법, 테스트 방법, 프로젝트 관리 기술 등이 발전하게 되었다.

    그에 따른 대표적인 기술은 SDLC, OOP, Pattern, Debug, Git 등이 있다.

    - 결론
    소프트웨어 위기는 규모과 복잡성이 증가함에 따라 발생한 개발 및 관리 문제를 의미하며, 지금의 소프트웨어 공학의 발전 및 체계적인 개발 방식의 기반이 됐다.


#### 6. SDLC
- 28 April, 2026
    - SDLC (Software Development Life-Circle)란 소프트웨어를 개발하는 전체 과정을 단계별로 정리한 개발 생명주기를 의미한다.

    이는 소프트웨어를 보다 체계적으로 개발 및 유지보수 하기 위해 사용된다.

    - 목적
    
    개발 과정의 체계화, 일정 및 비용관리, 품질 향상, 유지보수 효율 증가, 프로젝트 위험 감소 가 그 목적이며, 

    1. Requirement Analysis : 사용자가 원하는 기능과 목표 분석
    2. Design : 구조와 동작 방식 설계
    3. Implementation : 코드 구현
    4. Testing : 디버깅 하는 단계
    5. Deployment : 출시
    6. Maintenance : 유지보수

    총 6단계로 구성되어 있으며, 게임 개발에서의 SDLC 를 예시로 들자면 
    | 단계 | 예시 |
    | --- | --- |
    | 요구사항 분석 | 메트로베니아 게임 제작 결정 |
    | 설계 | 맵 구조, 전투 시스템 설계 |
    | 구현 | Unity 코드 작성 |
    | 테스트 | 버그 수정 및 밸런스 테스트 |
    | 배포 | Steam 출시 |
    | 유지보수 | 패치 및 콘텐츠 추가 |
    
    이렇게 되겠다.

    - 결론

    SDLC는 소프트웨어 개발 과정을 보다 체계적으로 관리하기 위한 생명주기 모델이며, 요구사항 분석부터 유지보수까지의 모든 과정을 단계별로 수행함으로써 소프트웨어 품질과 개발 효율을 향상시킬 수 있다.


#### 7.다향성 (Polymorphism)
- 29 April, 2026
    - 하나의 형태가 여러가지 방식으로 동작할 수 있는 객체지향 프로그래밍(OOP)의 특징이다.

    즉, 같은 메서드나 명령이라도 객체에 따라 서로 다르게 동작할 수 있다는 의미이다.

    - 특징
        - 같은 이름의 메서드 사용 가능.
        - 객체 종류에 따른 다른 결과값 출력됨.
        - 코드 재사용돠 유지보수성이 향상된다.

    - 예시
    - 동물 클래스 :
    
            class Animal 
            {
                public virtual void Sound()
                {
                    Debug.Log("동물 소리");
                }
            }


    - 강아지 클래스 : 
            class Dog : Animal
            {
                public override void Sound()
                {
                    Debug.Log("멍멍");
                }
            }

    - 고양이 클래스 : 
            class Cat : Animal
            {
                public override void Sound()
                {
                    Debug.Log("야옹");
                }
            }

이렇게 세개의 각자 클래스가 있다. 이때, 실행되는 함수는

        Animal animal;

        animal = new Dog();
        animal.Sound();

        animal = new Cat();
        animal.Sound();

이런식으로 함수가 정해지며, 그에따른 실행 결과는 멍멍 또는 야옹 으로 출력이 될 것이다.

이 말은 즉, 같은 Sound() 매서드를 호출했지만 객체 종류에 따라 다른 결과라 출렫된다는 걸 알 수 있고 우리는 이를 다형성 이라고 한다.


- 장점
1. 공동된 코드를 부모 클래스에서 관리할 수 있어, 코드 재사용성이 증가한다.
2. 기능 수정 시, 부모 클래스 또는 특정 클래스만 수정하면 돼, 유지 보수에 용이하다.
3. 새로운 클라스 추가가 쉬워짐으로 인해, 확장성이 증가한다.

이를 게임 개발에서 활용할 때는 Unity 게임 개발에서는 다형성이 매우 자주 사용된다.
예를 들면, Enemy, Boss, Weapon, Item, Skill 등과 같은 곳에서 주로 사용되며, 공통된 부모 클래스를 상속받아 서로 다른 방식으로 동작한다.

- 결론
다형성은 하나의, 메서드나 객체가 상황에 따라 여러 형태로 동작하는 객체지향 프로그램의 핵심 특징이며, 이를 통해 코드의 재사용과 유지보수 및 확장을 향상시킬 수 있다.



#### 8. 가상화 (virtualization)
- 30 April, 2026
    - 하나의 물리적인 컴퓨터 자원을 여러 개의 가상 환경으로 나누어 사용하는 기술이다.
    즉, 실제 하드웨어 한 대 안에, 여러개의 가상 컴퓨터(Virtual Machine)를 실행할 수 있도록 만드는 기술이다.

    - 목적
    1. 하드웨어 자원 효율 증가
    2. 시스템 관리 효율 증가
    3. 테스트 환경 구축
    4. 비용 절감
    5. 보안 및 안정성 향상

    - 가상머신 (VM)

    가상화 기술을 통해 만들어진 가상의 컴퓨터를 VM(Virtual Machine)이라고 부르며 이는 운영체제 (OS), 메모리 (RAM), 저장공간 (Storage) 등을 독립적으로 사용하는 것처럼 동작한다.

    - 예시
        - Mac PC 안에서 Windows 와 Linux 실행이 가능하다.
        이 말은 즉, 하나의 PC 에서 여러 운영체제를 동시에 운영할 수 있다는 말이된다.

    - 장점
    1. 자원 효율 증가
    2. 테스트 환경 구축
    3. 안정성 향상
    4. 비용 절감

    - 단점
    1. 실제 하드웨어보다 성능 저하 문제 발생할 수 있다.
    2. 많은 메모리와 저장 공간이 필요하다.
    3. 설정 및 관리가 복잡할 수 있다.

    이에 따른 대표적인 가상화 프로그램으로는 VMware, VirtualBox, Hyper-V, Docker(컨테이너 기반의 가상화) 가 되겠다.

    이를 게임에서 활용할 때는, 
    게임 개발 서버에서는 서버를 테스트 하고, 빌드 환경 구성 과 운영체제의 호환성을 테스트 하는 등 가상화 기술이 사용되고 있다.

    - 결론
    가상화는 하나의 물리적인 컴퓨터 자원을 여러 개의 가상 환경으로 나누어 사용하는 기술이다.
    이를 통해 자원 활용과 관리 효율을 높일 수 있으며, 다양한 테스트 및 서버 환경 구축에 활용된다.
